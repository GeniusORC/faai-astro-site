## 1. 项目回顾与关键技术决策

### 1.1 背景
为建设「福建省人工智能学会·企业工作委员会」官网，前期规划（根目录 `企工委网站建设方案.md`、`企工委网站_执行计划.md`）推荐 **Halo 2.x + PostgreSQL + Docker** 方案。实施过程中，结合「开源、自托管、无 SaaS 订阅费、国内访问快、微信体验好」的核心诉求，决策转向 **Astro 静态站** 方案。

### 1.2 Halo → Astro 决策与理由
| 维度 | Halo 方案 | Astro 方案（采用） |
| --- | --- | --- |
| 成本 | 需服务器/数据库，潜在订阅或运维费 | 纯静态托管免费档；Worker 免费；无 SaaS 费 |
| 运维 | Docker + 数据库，较重 | 无服务端，零运维 |
| 国内/微信 | 需自行优化 | 纯静态 + HTTPS，微信内体验好 |
| 内容维护 | Halo 后台 | Decap CMS 写回 Git，同样非技术可维护 |
| 风险 | Halo PAT 权限坑（insufficient_scope） | 技术栈轻、可控 |

> **结论**：采用 **Astro 5（SSG）+ Decap CMS + Cloudflare Worker+KV**。原 Halo 规划文档标记废弃，保留备查。

### 1.3 技术栈落地
- 前端：Astro 5（唯一运行时依赖），手写 CSS（学术蓝 `#1a3a6e` / 中国红 `#c8102e`），移动端优先。
- 内容：Decap CMS（MIT，自托管）连 GitHub 仓库 `GeniusORC/faai-astro-site`，写回 `src/content/notices/`。
- 表单：Cloudflare Worker + KV（`worker/`），原生表单 POST，微信友好，无第三方收费。
- 部署：EdgeOne / Cloudflare Pages / Vercel / Netlify 任选。

## 4. 经验与建议
- **轻量优先**：纯静态 + 边缘函数即可满足官网全部动态需求，显著降低运维与成本。
- **免费自托管可行**：Decap + Worker + KV 形成完全免费、数据自有的内容/表单闭环。
- **微信兼容**：纯 CSS 交互（无 JS 菜单）、原生表单提交，是微信内置浏览器稳定的关键。
- **文档先行**：本次补齐需求/架构/部署/用户/测试/运维/总结七件套，利于交接与持续维护。
- **备案与 HTTPS 前置**：国内访问与微信分享对 HTTPS 强依赖，应尽早完成。
